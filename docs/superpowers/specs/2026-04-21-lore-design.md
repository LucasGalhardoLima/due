# Lore — Design Spec

**Date:** 2026-04-21
**Status:** Approved

---

## Context

Lore is a personal knowledge OS built as a native Apple app. The problem: years of content saved across Threads, LinkedIn, Capacities, voice notes, and browser tabs — too scattered and high-friction for Claude to help with. Lore solves this by being the single place everything goes, with Foundation Models auto-organizing it, and an MCP server making the whole vault queryable by Claude.

New greenfield project at `~/Projects/lore/`. No dependency on any existing codebase.

---

## Product Identity

**"Lore"** in the sense of background knowledge and world-building — the accumulated context of everything you care about. Every article, post, and link you save is an entry in your personal codex. The aesthetic is a scholar's library: warm, atmospheric, dark-mode-forward but not enforced, clean and frictionless. RPG tone lives in copy and palette — not in decorative clutter.

---

## Core Philosophy

- **Offline-first.** Works with no account, no internet.
- **Privacy by default.** Everything runs on-device. No data leaves the device without explicit opt-in.
- **You just dump things in.** The AI sorts it all.
- **iCloud sync is optional / premium.**
- **AI-native.** MCP server makes the entire vault queryable by Claude from day one.

---

## Phase 1 Scope

Link vault only. iOS + macOS Share Extension → enrich on-device → searchable bento grid + MCP server. Notes come in Phase 2.

## Tech Stack

| Layer | Choice | Notes |
|---|---|---|
| Language | Swift 6.0+ | Strict concurrency |
| UI | SwiftUI | iOS + macOS universal |
| Persistence | SwiftData + App Groups | Shared container between app + extension |
| AI classification + summary | Apple Foundation Models | iOS 26+, macOS 15+; degrades gracefully |
| Semantic search | NLEmbedding | On-device, private |
| Link metadata | LPMetadataProvider | Native, zero external dependency |
| Content extraction | WKWebView + Readability.js | Bundled script, handles SPAs |
| Background jobs | BGProcessingTask | iOS-sanctioned background enrichment |
| MCP server | Node.js | Reads local SQLite export |
| iCloud sync | CloudKit via SwiftData | Phase 4, stubbed in LoreSync |

**Zero external services.** No jina.ai. No Supabase. No third-party network calls after the initial URL save.

## Data Models (LoreCore)

```swift
@Model class LoreEntry {
    var id: UUID
    var url: String
    var title: String
    var content: String?          // extracted by WKWebView + Readability.js
    var description: String?      // from LPMetadataProvider
    var heroImagePath: String?    // cached in app container
    var dominantColor: String?    // hex, card background fallback
    var aiSummary: String?        // 1-line, generated by Keeper
    var source: String            // "threads", "safari", "linkedin", "reddit"
    var savedAt: Date
    var status: CanonStatus       // .discovered | .canon
    var volume: Volume?           // assigned by Keeper; nil = unclassified
    var embedding: Data?          // NLEmbedding vector
    var tags: [String]
}

enum CanonStatus {
    case discovered   // stub — URL + title only, enrichment pending
    case canon        // fully enriched
}

enum Volume {
    case projects     // active things with a goal or deadline
    case areas        // ongoing responsibilities
    case resources    // reference material, saved links
    case archive      // dormant or completed
}
```

## Services

### Keeper (LoreAI)
Foundation Models wrapper. Input: title + description + content excerpt. Output: `aiSummary` (1 line) + `Volume` assignment. Degrades gracefully on pre-iOS 26 devices — entry stays visible, `volume` stays nil.

### Scribe (main app `Services/Scribe.swift`)
Enrichment pipeline. Triggered by `BGProcessingTask` or on app foreground.

1. Fetch all `.discovered` entries
2. `LPMetadataProvider` → description + thumbnail URL
3. Download + cache hero image locally; extract dominant hex color
4. `WKWebView` + Readability.js → full article text
5. `NLEmbedding` → semantic vector from title + content
6. Keeper → `aiSummary` + `Volume`
7. Flip `status` to `.canon`

**Failure handling:** any step failure leaves entry at `.discovered`. Scribe retries automatically next run. Nothing is lost, nothing is silent.

## Capture Flow

### Share Extension (iOS + macOS)
```
User taps Share → selects Lore
  → Extension reads NSItemProvider (public.url + public.plain-text)
  → Writes LoreEntry(.discovered) to shared SwiftData store
     (App Groups: group.com.lore.app)
  → "Added to your Codex" + tome icon animation
  → Dismisses in < 1 second
```
The extension never does enrichment. Stub write only.

### macOS paste URL
Persistent URL input field in the macOS app sidebar. Paste + return → same stub → Scribe pipeline.

### App Groups
Both the main app and `LoreShareExtension` targets declare `group.com.lore.app`. SwiftData store URL is derived from this shared container in both targets.

## MCP Server (Phase 1)

The macOS app writes a denormalized `lore.sqlite` export to `~/Library/Application Support/Lore/` on every `modelContext.save()`. The Node.js MCP server reads this file directly — no HTTP server, no app dependency, no live connection.

**MCP tools:**
| Tool | Description |
|---|---|
| `search(query)` | Semantic + keyword search across all entries |
| `get_entry(id_or_url)` | Full content of a specific entry |
| `list_volume(name)` | All entries in a given Volume |
| `list_recent(limit?)` | Most recently saved entries |

Claude queries the vault whether or not Lore is open.

## UI Structure

### Screens
- **Chronicle** — search hero at top, bento grid of all entries below
- **Entry detail** — hero image, title, aiSummary, full content, Volume badge, tags
- **Volumes** — entries filtered by PARA section (Projects / Areas / Resources / Archive)

### Navigation
- **iOS:** tab bar (Chronicle, Volumes, Settings)
- **macOS:** sidebar (Chronicle + Volumes + Settings), main content pane

### Bento Grid
Variable card sizes — image-rich entries get more space. Cards breathe with generous padding. Each card: hero image or dominant color fill, title, 1-line aiSummary. `.discovered` entries show a pulsing amber glow with "The Keeper is working..."

### Search
Semantic (NLEmbedding cosine similarity) + keyword LIKE fallback. Results replace the bento grid inline. Placeholder: *"Search your lore..."*

### Parchment (light)
| Token | Value |
|---|---|
| Background | `#F5F0E8` |
| Surface | `#EDE8DC` |
| Text primary | `#2C1F14` |
| Text secondary | `#7A6A56` |
| Accent | `#A0731A` (burnished gold) |
| Texture | Subtle parchment grain |

**Typography:** Serif display face for headings and card titles. Clean sans-serif for body and UI chrome. SF Symbols for system icons.

**Aesthetic principle:** Scholar's library — warm, atmospheric, never dense. The RPG identity lives in the palette, texture, and copy. Not in decorative clutter. Designed to scale cleanly when notes are added in Phase 2.

## Phase Roadmap

### Phase 1 — Link vault + MCP (this spec)
iOS + macOS Share Extension, on-device enrichment, bento grid Chronicle, search, Volume browsing, Grimoire + Parchment themes, MCP server.

### Phase 2 — Notes + adaptive cards
Typed notes and voice capture in the same Codex. Adaptive card style by content type. "Dusk" theme.

### Phase 3 — iCloud Sync (Premium)
CloudKit sync via SwiftData across iOS + macOS. Monetization tier.

### Phase 4 — Apple Intelligence
Siri App Intents, Foundation Models conversation UI, Action Button + Lock Screen Widget.
